# 华西区域暑期预警数据看板

一个由真实上传数据驱动的 React + TypeScript 经营预警驾驶舱，支持本地 Express 数据服务，以及 Netlify Functions + Netlify Blobs 线上统一版本存储。

## 启动

```bash
npm install
cp .env.example .env
npm run dev
```

开发环境：

- 只读看板：`http://localhost:5173/dashboard`
- 管理后台：`http://localhost:5173/admin`

当前交付版本按要求取消管理员密码，打开 `/admin` 即可进入后台。生产环境执行 `npm run build && npm start`，默认服务地址为 `http://localhost:8787`。

## 推荐上传顺序

本地服务版可读取预置基础表；Netlify 首次上线时按交付指南在后台依次导入酒店维度表、去年同期暑期经营表、同期同提前期预订快照表、改造店明细和当前快照。首次初始化后，日常只需上传“今年未来 7 天预订快照表”；维度或同期同提前期基准变化时，再覆盖更新对应基础表。

1. 酒店维度表 `DimHotel`：系统预置，低频维护
2. 去年同期暑期经营表 `LastYearSummer`：系统预置，作为暑期同期基准
3. 同期同提前期预订快照表 `SameLeadBookingSnapshot`：去年周对周同提前期开盘预订基准
4. 改造店明细 `Renovations`：按 WH 编码识别轻改 / 中改
5. 今年未来 7 天预订快照表 `FutureBookingSnapshot`：每日更新，最新文件整体覆盖

支持 `.xlsx`、`.xls`、`.csv`。页面会自动识别常见字段名；无法识别的字段可在保存前手动映射。一个快照文件中如包含多个“快照日期 + 快照批次时间”，会自动拆分并分别保留。

酒店维度表采用非阻断式数据质量校验：只要能够识别“酒店WH编码”和“酒店名称”表头，且至少存在一条同时包含这两个字段的有效记录，即可保存。商圈、物理房量、经营状态及其他维度字段缺失只进入数据警告；门店明细仍只展示经营状态为“在营 / 在营业”的门店。

## 数据保存机制

- 本地开发时，数据统一保存在服务端 `server/data/store.json`，不是 localStorage，也不是浏览器内存。
- Netlify 部署时，数据通过 Netlify Functions 分块写入 Netlify Blobs。
- 大型未来预订快照由浏览器自动拆成约 768KB 的数据块并逐块上传，后台会显示“正在保存 x/y”；管理员仍只需选择一次完整 Excel。
- 同期同提前期预订快照也使用相同的分块发布机制。
- `currentVersion` 只在全部数据块保存成功后更新；普通用户刷新页面时先读取 `currentVersion`，再读取该版本的数据清单和数据块。
- 线上仅保留当前版本、上一版本及其仍被引用的数据块，避免历史原始数据无限增长。
- `/dashboard` 只提供公共读取，不提供上传、覆盖或删除入口。
- `/admin` 当前为直接访问模式，不需要账号或密码；任何知道后台地址的人都可以修改线上数据，建议仅在内部网络或受控 Netlify 站点中使用。
- 建议在“数据上传中心”定期使用“导出数据备份”生成 JSON 备份。
- 可使用“导入备份恢复”恢复维度、同期、快照和映射配置。

## 部署到 Netlify

项目已包含 `netlify.toml` 和 `netlify/functions/api.mjs`。

1. 将项目部署到 Netlify，构建命令使用 `pnpm run build`，发布目录为 `dist`。
2. 当前直接访问版本无需配置 `ADMIN_PASSWORD` 或 `SESSION_SECRET`。
3. 首次部署后直接打开 `/admin`。如果线上 Blobs 还是空的，可在本地后台先“导出数据备份”，再在线上后台使用“导入备份恢复”，完成一次性迁移。
4. 首次迁移完成后，日常只需上传“未来7天预订快照表”。系统会自动：
   - 识别文件内最早目标日期作为基准日；
   - 看板首次打开默认选中该窗口第一天（D0）；
   - 覆盖当前版本快照，不追加重复跑批；
   - 保留上一版同目标日期末次快照；
   - 分块写入 Blobs；
   - 全部保存成功后更新 `currentVersion`；
   - 让所有看板用户刷新后读取线上最新版本。

由于当前完整数据约 193MB，系统不会将整份数据作为一个 Function 请求发送，而是拆分为多个小数据块，规避 Netlify Functions 单次请求体限制。

## 关键口径

- 同期优先使用指定映射日期；无映射时使用目标日期减 364 天。
- 渠道明细先按“酒店WH编码 + 年月日 + 跑批次数”汇总到门店级，可售房只取一次。
- 预订率为汇总预订房间数 / 去重可售房；在手 ADR 为汇总预订总价 / 汇总有价房间数；理论 RP 为汇总预订总价 / 去重可售房。
- 未来预订文件每次整体覆盖，当前跑批为文件内最大跑批，环比取同文件内上一跑批。
- 门店提价建议按 D0-D6 分层，并综合本店/商圈订单进速、同期空间、ADR差异、剩余库存及特殊商圈属性生成价格动作与简短提醒。
- 同期同提前期按周对周日期匹配，仅作为开盘预订进度比较，不与去年最终经营结果混淆；缺失时安全显示为空，不计入经营异常。

## 页面性能与大表保护

- 门店预警明细与渠道异常 TOP 默认仅渲染当前 20 行，可切换 10 / 20 / 50 行；导出仍覆盖当前筛选后的全部结果。
- 渠道异常等重分析模块只在进入对应 Tab 后挂载计算。
- 管理员发布版本时会额外生成公共看板精简数据集：按“门店 + 真实目标日期 + 标准渠道组”预聚合，并只保留当前 D0-D6 所需的去年同期和同提前期记录。
- 普通用户首次打开 `/dashboard` 优先读取精简数据集；进入完整渠道分析视图时再按需读取原始一级/二级/三级渠道数据。管理员后台始终读取完整原始版本。
- 线上原始数据使用内容寻址的不可变小块保存；上传失败不会切换 `currentVersion`，普通用户继续读取上一完整版本。
- 构建产物把图表、Excel 与 React 依赖拆成独立缓存块，降低后续页面更新下载量。

## 地图下钻

- 首页使用真实行政区 GeoJSON，不再使用手绘示意轮廓。
- 华西总览点击省份后进入该省真实城市边界。
- 城市地图点击城市后联动左侧城市筛选及全页面指标。
- 点击地图内“返回华西总览”可返回省区总览。
