# 华西区域暑期预警数据看板：Netlify 上线与日常更新完整指南

本指南对应根目录可直接上传 GitHub 的完整源码包。照顺序操作即可完成：

- 公共看板上线；
- 管理后台直接访问；
- 酒店维度、同期经营、改造店和未来预订数据初始化；
- 管理员每日上传最新快照；
- 省区、片区用户刷新网页后读取同一份线上最新数据。

---

## 一、交付包结构

解压 `west-china-summer-warning-netlify-full.zip` 后，根目录直接看到：

```text
package.json
pnpm-lock.yaml
pnpm-workspace.yaml
netlify.toml
src/
public/
server/
netlify/functions/api.mjs
README.md
deployment/
```

注意：

1. 将解压后的根目录内容全部上传到 GitHub，仓库根目录必须直接看到 `package.json` 和 `netlify.toml`。
2. 压缩包不包含 `.env`、`node_modules`、本地数据备份和私密配置。
3. 当前版本按要求取消后台登录密码，无需配置密码环境变量。
4. 不要仅把 `dist` 文件夹拖到 Netlify。这样只有静态页面，后台上传接口和线上数据存储都不会工作。

---

## 二、推荐部署方式

推荐使用：

```text
私有 GitHub 仓库 + Netlify 连续部署 + Netlify Functions + Netlify Blobs
```

采用此方式后：

- 页面和函数由 Netlify 一起部署；
- `/admin` 按当前要求直接访问；
- 上传数据保存在该 Netlify 项目的 Blobs 中；
- 以后重新部署代码不会清空经营数据；
- 所有省区、片区用户访问同一个网址，读取同一个 `currentVersion`。

Netlify 官方说明：

- Functions 部署：https://docs.netlify.com/build/functions/get-started/
- 环境变量：https://docs.netlify.com/build/environment-variables/get-started/
- Netlify Blobs：https://docs.netlify.com/build/data-and-storage/netlify-blobs/

## 三、第一步：建立私有 GitHub 仓库

### 1. 登录 GitHub

打开：

```text
https://github.com/
```

### 2. 新建仓库

点击右上角 `+` → `New repository`。

建议填写：

```text
Repository name: west-china-summer-warning
Visibility: Private
```

一定选择 `Private`，不要建立公开仓库。

### 3. 上传部署源码

进入新仓库后，点击：

```text
Add file → Upload files
```

上传解压后根目录里面的全部内容。

注意根目录必须直接看到：

```text
netlify.toml
package.json
src
public
netlify
server
```

不要把 `01-网站部署源码` 作为额外一层目录上传，否则 Netlify 找不到项目根目录。

提交说明可填写：

```text
Initial Netlify deployment
```

然后点击 `Commit changes`。

---

## 四、第二步：在 Netlify 创建网站

### 1. 登录 Netlify

打开：

```text
https://app.netlify.com/
```

推荐使用刚才的 GitHub 账号登录。

### 2. 导入 GitHub 项目

在 Netlify 中选择：

```text
Add new project / Add new site
→ Import an existing project
→ GitHub
```

授权后选择刚才建立的私有仓库：

```text
west-china-summer-warning
```

### 3. 检查构建设置

项目中的 `netlify.toml` 已经写好配置，正常情况下 Netlify 会自动识别：

```text
Build command: pnpm run build
Publish directory: dist
Functions directory: netlify/functions
Node version: 20
```

如果页面要求手工填写，按上面内容填写。

然后点击部署。

第一次部署完成后即可继续下一步，无需设置管理员密码环境变量。

## 五、第三步：确认直接访问后台

当前版本按要求取消 `/admin` 登录密码，不需要在 Netlify 设置 `ADMIN_PASSWORD` 或 `SESSION_SECRET`。

部署完成后直接打开：

```text
https://your-project-name.netlify.app/admin
```

即可进入数据上传中心。需要重新发布代码时，进入：

```text
Deploys → Trigger deploy → Deploy site
```

重新部署一次，让环境变量对 Functions 生效。

## 七、第五步：首次初始化线上数据

首次部署后，Netlify Blobs 是空的。只需要初始化一次。

打开：

```text
https://your-project-name.netlify.app/admin
```

登录后进入“数据上传中心”，严格按以下顺序上传。

### 1. 酒店维度表

选择：

```text
酒店维度表
```

上传：

```text
02-首次初始化数据/01-酒店维度表.xlsx
```

操作：

1. 点击“选择文件”；
2. 检查字段映射；
3. 点击“执行校验”；
4. 确认没有阻断性错误；
5. 点击“校验并保存”；
6. 等待出现“上传成功”。

### 2. 去年同期暑期经营数据

选择：

```text
去年同期暑期经营表
```

上传：

```text
02-首次初始化数据/02-去年同期暑期经营数据.xlsx
```

同样执行校验并保存。

这个文件只需首次上传。除非同期基准数据发生变化，否则日常不用重复上传。

### 3. 同期同提前期预订快照表

选择：

```text
同期同提前期预订快照表
```

上传去年周对周同提前期的开盘预订数据。必需可识别字段为：

- 酒店WH编码；
- 年月日；
- 预订房间总数；
- 预订总价；
- 可售房间数。

可售房为空或 0、酒店名称为空、有房价预订房间数为空、WH 未匹配维度表只提示 warning，不阻断整表保存。该表按周对周日期匹配，例如今年 `2026-07-02` 对应去年 `2025-07-03`。

### 4. 改造店明细

选择：

```text
改造店明细
```

上传：

```text
02-首次初始化数据/03-改造店明细.xlsx
```

字段为：

- 酒店WH编码；
- 改造类型。

校验并保存后，门店视图中的轻改、中改和改造店专项才会生效。

### 5. 当前未来7天预订快照

选择：

```text
未来7天预订快照表
```

上传：

```text
02-首次初始化数据/04-未来7天预订快照_20260630.xlsx
```

校验并保存。

该文件原始日期范围是：

```text
2026-06-30 至 2026-07-07
```

首页默认展示前7天：

```text
D0 06-30
D1 07-01
D2 07-02
D3 07-03
D4 07-04
D5 07-05
D6 07-06
```

保存成功提示中必须显示：

```text
D0 已更新为 2026-06-30
```

如果仍显示 06-29，说明部署的不是本次交付版本，或浏览器仍在使用旧部署。

### 首次初始化成功标志

后台顶部应显示：

- 门店数量大于0；
- 同期记录大于0；
- 改造记录大于0；
- 快照批次大于0。

公共看板顶部应显示：

- 数据来源：线上最新版本；
- 当前版本号；
- 数据更新时间；
- 当前跑批；
- 覆盖日期；
- 正确的 D0-D6 日期。

## 八、以后每天如何更新最新数据

首次初始化完成后，日常只需要上传：

```text
未来7天预订快照表
```

操作路径：

```text
/admin
→ 数据上传中心
→ 未来7天预订快照表
→ 选择文件
→ 执行校验
→ 校验并保存
```

### 每日三次跑批

建议对应：

```text
跑批1：10:20
跑批2：14:20
跑批3：21:20
```

每次应上传“截至当前时点的完整文件”。

例如第二次文件包含跑批1和跑批2，直接上传完整文件；系统会整体覆盖当前版本，不会把跑批1重复追加。

第三次文件包含跑批1、跑批2、跑批3，也上传完整文件。

### 上传后的系统动作

管理员点击“校验并保存”后，系统会：

1. 从文件“年月日”中识别最早日期作为 D0；
2. 重新生成 D0-D6；
3. 按跑批拆分当前版本；
4. 同基准日重复上传时整体覆盖；
5. 日期窗口变化时保留上一版同目标日期的末次快照；
6. 将新数据分块写入 Netlify Blobs；
7. 所有数据块成功后才更新 `currentVersion`；
8. 普通用户刷新 `/dashboard` 后读取最新版本。

上传过程中不要关闭页面。只有看到“上传成功”才代表 `currentVersion` 已切换。

## 九、省区、片区用户如何查看最新数据

把公共地址发给用户：

```text
https://your-project-name.netlify.app/dashboard
```

普通用户：

- 不需要登录；
- 看不到数据上传中心；
- 看不到预警播报后台；
- 不能修改或删除数据；
- 可以使用省区、片区、城市、商圈、门店和渠道筛选、下钻、排序与导出。

管理员上传成功后，用户刷新页面即可看到最新数据。

建议通知语：

```text
最新跑批数据已更新，请刷新看板页面查看。
```

如果用户浏览器仍显示旧数据：

1. 先普通刷新；
2. 再使用强制刷新：
   - Windows：`Ctrl + F5`
   - Mac：`Command + Shift + R`
3. 查看顶部“当前版本号”和“数据更新时间”是否已经变化。

## 十、为什么所有用户都能看到同一份最新数据

线上数据不保存在管理员浏览器的 localStorage，也不保存在个人电脑中。

保存链路为：

```text
管理员浏览器
→ /api/admin/data/chunk
→ Netlify Function
→ Netlify Blobs 保存数据块
→ /api/admin/data/publish
→ 更新 currentVersion
→ 普通用户读取 /api/data
→ 加载 currentVersion 对应数据
```

`currentVersion` 只有在全部数据块成功保存后才切换，因此上传中途失败不会让普通用户读到半份数据。

Netlify Blobs 使用站点级存储，代码重新部署后仍会保留。

## 十二、数据备份与恢复

建议每天第三次跑批上传成功后：

```text
/admin
→ 数据上传中心
→ 导出数据备份
```

将 JSON 备份保存到公司受控目录。

建议命名：

```text
华西暑期预警备份_2026-06-30_跑批3.json
```

需要恢复时：

```text
/admin
→ 导入备份恢复
→ 选择备份JSON
```

注意：

- Netlify 的代码回滚只回滚页面和 Functions；
- Blobs 中的经营数据不会跟随代码部署回滚；
- 数据误传时应使用后台备份恢复。

## 十三、维度表和同期数据何时更新

### 酒店维度表

平时不用每天上传。仅在以下情况更新：

- 新增门店；
- 门店停业或经营状态变化；
- 省区、片区、城市、行政区、收益管理商圈调整；
- 品牌、定位、经营类型、管理类型变化；
- 开业日期修正。

更新方式是上传完整维度表整体覆盖。

### 去年同期暑期经营数据

原则上暑期开始前初始化一次。

只有在同期经营底表补数或修正时才重新上传完整文件。

### 改造店明细

有新的轻改、中改门店时再上传完整明细。

### 未来7天预订快照

每天按跑批更新，是日常主要操作。

## 十四、常见问题排查

### 1. `/admin` 无法打开

确认完整源码包已经通过 GitHub 部署，且 Netlify 发布目录为 `dist`。当前版本无需设置管理员密码环境变量。

### 2. `/api/data` 返回网页HTML或404

说明 Functions 没有部署。

常见原因：

- 只拖拽了 `dist`；
- `netlify.toml` 不在仓库根目录；
- `netlify/functions/api.mjs` 未上传；
- Netlify 构建失败。

处理：

- 改用 GitHub 连续部署或 Netlify CLI；
- 查看 Netlify 的 Deploy log 和 Functions log。

### 4. 网站打开但没有数据

说明线上 Blobs 尚未初始化。

按本指南第七部分依次上传四类数据。

### 5. 上传后普通用户仍看到旧数据

检查后台是否显示“上传成功”，再比较：

- 当前版本号；
- 数据更新时间；
- 当前跑批；
- 覆盖日期。

让用户强制刷新页面。

### 6. D0 日期提前一天

本交付版本已经修复 Excel 日期时区偏移。

请确认：

- 部署的是本次交付包；
- 上传文件“年月日”列真实最早日期正确；
- 上传成功提示中的 D0 正确。

### 7. 上传中断

重新登录后台并重新上传同一完整文件。

`currentVersion` 未切换前，普通用户仍会继续读取上一完整版本。

### 8. Netlify 构建失败

核对：

```text
Build command: pnpm run build
Publish directory: dist
Node version: 20
```

并查看 Deploy log 中第一条红色错误。

### 9. 使用量或费用异常

当前数据量较大，建议在 Netlify 控制台定期查看：

- Functions 使用量；
- Bandwidth；
- Blobs 存储和读取；
- 构建分钟数。

项目会自动清理不再被当前版本和上一版本引用的数据块，避免历史数据无限增长。
